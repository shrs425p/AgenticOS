## Objective
Integrate command safety checks into the terminal subprocess runner and script execution pipelines, formatting validation failures uniformly and logging security alerts in the SQLite audit database.

<task type="auto">
  <name>Task 2: Implement line-by-line script safety validation in run_script</name>
  <files>tools/terminal/runner.py</files>
  <read_first>tools/terminal/runner.py</read_first>
  <action>
    Update `run_script` in `tools/terminal/runner.py` to scan interior lines of shell scripts before launching.
    - Check if the script file extension/suffix matches `.ps1`, `.bat`, `.cmd`, `.sh`, or `.bash`.
    - Read the file content safely with utf-8 encoding and character replacement (`errors="replace"`).
    - Remove comments: comments start with `#` in `.sh`, `.bash`, `.ps1` and `REM ` (case-insensitive) or `::` in `.bat`, `.cmd`.
    - Handle line continuation merges: merge lines ending in `\` (POSIX), `` ` `` (PowerShell), or `^` (CMD/Batch).
    - Run each reconstructed command line through `self._blocked_command_reason(cmd)`. If blocked, return the normalized `f"Error: {reason}"` string.
  </action>
  <verify>.\\venv\\Scripts\\pytest tests/test_terminal_safety_structural.py</verify>
  <acceptance_criteria>
    - Scripts with suffix .sh, .bash, .ps1, .bat, or .cmd containing blocked commands are intercepted and return the normalized error.
    - Comments, blank lines, and line continuations are handled correctly.
    - Python (.py) scripts are skipped from line-by-line command scanning.
  </acceptance_criteria>
  <done>Line-by-line shell script validation is implemented.</done>
</task>
