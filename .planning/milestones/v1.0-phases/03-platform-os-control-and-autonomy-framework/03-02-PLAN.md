<threat_model>
## Trust Boundaries
- File system read/write boundary for storing checkpoint JSONs and SQLite database under `workspace/.checkpoints/`.
- System profiling boundary (psutil API calls) retrieving host resource metadata.

## STRIDE Register
| ID | Threat | Component | Mitigation |
|----|--------|-----------|------------|
| T-03-04 | Information Disclosure | checkpoint_manager.py | Ensure checkpoints do not store raw credentials/secrets; SQLite connection must restrict access permissions to the running user. |
| T-03-05 | Denial of Service (OOM) | resource_profiler.py | Proactively monitor memory usage; dynamically scale down worker execution thread count if available RAM drops below 20%. |
</threat_model>

## Artifacts this phase produces
- `kernel/resources.py`: System profiler defining resource thresholds, scaling recommendations, and memory pressure checks.
- `kernel/checkpoint.py`: Linear checkpoint manager with goal hashing and dual JSON/SQLite storage.
- `spec/resourceprofilerspec.py`: Test suite with mocked psutil resource levels.
- `spec/checkpointmanagerspec.py`: Test suite for JSON and SQLite checkpoint persistence.

  <read_first>kernel/resources.py, .planning/phases/03-platform-os-control-and-autonomy-framework/03-RESEARCH.md</read_first>
  <action>
    Create `kernel/resources.py`.
    - Define `HardwareProfile` dataclass containing: `ram_tier`, `cpu_count`, `available_ram_gb`, `total_ram_gb`, `recommended_max_workers`, `recommended_context_tokens`, `recommended_cache_mb`, `compact_history_threshold`.
    - Define three RAM tiers:
      - Low (<= 4 GB): workers=2, context=8192, cache=128, compact=15.
      - Mid (> 4 GB and <= 16 GB): workers=4, context=32768, cache=512, compact=30.
      - High (> 16 GB): workers=cpu_count, context=131072, cache=2048, compact=60.
    - Implement `profile_hardware() -> HardwareProfile` querying psutil virtual_memory, cpu_count, and disk_usage.
    - Implement `check_memory_pressure(current_workers: int) -> int` to halve max_workers if virtual memory usage is above 80%.
  </action>
  <verify>Review resource_profiler.py structure to ensure all tiers are strictly implemented without placeholders.</verify>
  <done>resource_profiler.py is created and fully implemented.</done>
</task>

<task type="auto">
  <name>Task 2: Implement kernel/checkpoint.py checkpoint loops</name>
  <files>kernel/checkpoint.py</files>
  <read_first>kernel/checkpoint.py, .planning/phases/03-platform-os-control-and-autonomy-framework/03-RESEARCH.md</read_first>
  <action>
    Create `kernel/checkpoint.py`.
    - Implement `_goal_to_task_id(goal: str) -> str` using SHA256 hashing (strip + lowercase before hashing) truncated to 12 chars.
    - Implement `CheckpointManager` class with JSON file storage under `workspace/.checkpoints/` and SQLite secondary DB under `workspace/.checkpoints/checkpoints.sqlite3`.
    - Implement methods:
      - `create(goal, phases) -> str`: creates task checkpoint, inserts SQLite metadata entry, saves JSON file.
      - `load(goal) -> dict | None`: loads checkpoint dict by goal hash from JSON.
      - `update_phase(task_id, phase_name, status, result=None) -> None`: updates specific phase status ("pending", "running", "complete") in JSON and SQLite DB.
      - `next_pending_phase(task_id) -> dict | None`: returns first phase where status != "complete".
  </action>
  <verify>Ensure checkpoint manager handles file paths properly and manages SQLite connections safely.</verify>
  <done>checkpoint_manager.py is created and SQLite/JSON dual persistence is fully implemented.</done>
</task>
